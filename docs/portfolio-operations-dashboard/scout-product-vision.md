# Scout Product Vision

## Product Intent

Scout is not a chat feature attached to ATLAS.

Scout is the intelligence engine inside ATLAS that continuously monitors portfolio performance, prioritizes operational risks, surfaces opportunities, and guides teams toward the highest-impact actions.

In its most powerful form, Scout becomes the AI COO for the portfolio.

## UI / UX Vision

Scout should feel:

- clean
- intelligent
- fast
- operational
- predictive

Reference direction:

- more like Palantir
- more like Notion AI
- more like Monday Intelligence
- less like Entrata reporting screens

## Experience Standard

When a user opens Scout, the experience should feel:

- focused, not cluttered
- high-signal, not report-heavy
- decisive, not vague
- contextual, not generic
- premium, not utilitarian

Scout should not feel like:

- a generic support chatbot
- an embedded novelty feature
- a static reporting pane
- a passive search box

## Product Role

Scout's role inside ATLAS is to:

- identify problems
- predict issues
- recommend actions
- eventually automate workflows

This is the progression from assistant to operating system layer.

## Product Evolution

### Phase 1: Explain

Scout explains what changed, what is driving it, what matters most, and what cannot be confirmed from the current data.

Primary behaviors:

- summarize
- rank risk
- explain variance
- recommend next checks
- suggest actions
- draft follow-up talking points

### Phase 2: Guide

Scout becomes a proactive operating partner.

Primary behaviors:

- continuously monitor key metrics
- surface exceptions automatically
- benchmark similar assets
- identify likely root causes from ATLAS data patterns
- suggest interventions before the next review cycle

### Phase 3: Orchestrate

Scout becomes the portfolio intelligence and workflow engine.

Primary behaviors:

- create follow-up tasks
- notify regionals or leaders
- route issues to teams
- draft playbooks or meeting briefs
- trigger approved automations

### Phase 4: Operate

Scout becomes the AI COO layer for ATLAS.

Primary behaviors:

- monitor portfolio health continuously
- prioritize actions dynamically
- coordinate workflows
- recommend the next highest-impact decisions
- automate approved operating loops with human oversight

## Long-Term Workflow Vision

Example:

Scout detects:

- declining conversion at GKP

Scout then:

- flags the issue
- identifies the likely root cause
- suggests a pricing adjustment
- creates follow-up tasks
- notifies the regional
- benchmarks against similar assets

This is the transition point where ATLAS stops being software and becomes an operational command center.

## Internal Positioning

Internal narrative:

`Scout is the intelligence engine inside ATLAS that continuously monitors portfolio performance, prioritizes operational risks, surfaces opportunities, and guides teams toward the highest-impact actions.`

This is the core internal product positioning and should remain stable across future feature expansion.

## Operating Principles

Scout should always:

- stay grounded in trusted ATLAS data
- prioritize signal over volume
- surface risk before users ask
- make the next action obvious
- adapt to VP and Regional decision needs
- preserve trust through honesty and clarity

Scout should never:

- flood the user with dashboards
- make ungrounded claims
- automate sensitive actions without clear approval boundaries
- feel ornamental

## UI Design Principles

### 1. Command Center, Not Chat Toy

The UI should frame Scout as an operating layer, not just a messaging panel.

Design implications:

- highlight active risks and opportunities before the input field
- show ranked priorities
- make the current scope obvious
- use structured cards, status bands, and action panels

### 2. Intelligence Before Conversation

Scout should show useful intelligence even before the user types.

Design implications:

- default risk feed
- top changes this period
- communities needing attention
- watchlist items
- unresolved data trust issues

### 3. Fast Interpretation

Scout should help users grasp the situation within seconds.

Design implications:

- strong hierarchy
- concise summaries
- expandable detail
- minimal decorative noise

### 4. Context Is Everything

Scout should always know and display:

- current portfolio or community scope
- current period
- latest source freshness
- confidence level
- ranking basis

### 5. Action Loop Closure

Scout should not stop at explanation.

Design implications:

- each major insight should connect to a next check, action, or follow-up path
- action recommendations should feel native to ATLAS workflows

## V1 UI Direction

In version one, Scout should present:

- a high-priority summary area
- ranked risks
- one concise narrative summary
- suggested next checks
- suggested actions
- follow-up talking points

The input area should remain present, but the product should not rely on the user to ask the perfect question.

## V2 UI Direction

In version two, Scout should add:

- watchlists
- proactive alerts
- benchmark views
- risk trend movement
- role-specific daily briefings

## V3 UI Direction

In version three, Scout should add:

- workflow triggers
- automation approvals
- task creation
- escalation routing
- operating cadences and briefing generation

## Success Standard

Scout is succeeding when users say:

- "It told me where the risk is."
- "It helped me know where to focus."
- "It caught the issue before the meeting."
- "It gave me the next step, not just the data."
- "It feels like an operating partner."

## Strategic Outcome

The strategic goal is not to add AI to ATLAS.

The strategic goal is to make ATLAS the operating command center for portfolio decision-making, with Scout as the intelligence layer that makes the system proactive, prioritized, and increasingly autonomous.
